# 企业 ESG 产品闭环连续执行计划

**执行状态：** 阶段 0-8 已于 2026-07-11 完成全部自动门禁，当前正常停止在人工产品验收点。阶段门禁产物位于 `tmp/product-closure-continuous/`；该目录是本地临时验收证据，不作为长期文档源。

## 1. 目标

本计划连续执行 `docs/plan/product-closure-implementation-plan.md` 已定义的全部 8 个纵向阶段。阶段 1、2、3 是前置阶段，不设置固定人工停止点。只要自动门禁通过且未触发停止条件，就继续执行下一阶段，直至完成阶段 8 并生成最终人工验收包。

产品目标保持为：面向企业 ESG 团队，完成报告上传、577 条分析、风险复核、整改和版本化导出的单报告闭环。普通界面不暴露 profile、ontology、route、evidence kind 等内部实现字段。

## 2. 执行边界

- 在 `main` 分支 inline 执行，不创建新分支或 worktree。
- 不调用外部模型，不启用 OCR/VLM，除非用户另行明确批准。
- 不覆盖、不回退当前工作区中与本计划无关的改动。
- 每个行为变更先写失败测试，再实现最小代码，再运行相关测试。
- 阶段之间不等待人工确认；自动门禁通过后继续。
- 不自动 commit 或 push。全部阶段完成或触发停止条件时，整理变更和验证结果交给用户。
- 临时验证产物写入 `tmp/product-closure-continuous/`，不提交人工报告或原始来源资产。

## 3. 连续执行顺序

### 阶段 0：基线与执行准备

1. 读取 `README.md`、`docs/DESIGN.md`、`docs/DEVELOPMENT.md`、`docs/ASSETS.md` 和本计划。
2. 记录工作区现有改动，不将脏工作区本身视为停止条件。
3. 启动本地 PostgreSQL，检查 Alembic 当前 revision 和现有数据量。
4. 运行后端、前端和 Envision 577 基线测试；记录已存在失败。
5. 建立阶段验收记录，保存测试命令、结果、migration revision 和兼容周期计数。

**门禁：** 能区分已有失败与本计划引入的失败；数据库可备份和迁移；577 eligible requirement 数量保持不变。

### 阶段 1：报告列表与 metadata 确认

按 `product-closure-implementation-plan.md` 阶段 1 实施 migration、报告列表、重复上传保护、metadata 确认和分析启动门禁。

**验收：** 上传、列表、确认、重复哈希 409、未确认禁止分析；后端测试、OpenAPI 生成、前端测试和 build 全部通过。

### 阶段 2：577 条分析进度与部分失败

实施完整 577 task、七阶段进度、requirement 级失败隔离和失败项重跑。生产 API 不保留 10 条限制，测试 fixture 可显式限制。

**验收：** 成功结果在部分失败时保留；retry 只处理失败项；进度事件只追加；分析可在离开页面后继续。

### 阶段 3：固定风险模型与队列

实施 `risk-v1`、风险原因代码、dashboard 汇总、分页 assessment 和高风险复核队列。风险规则使用固定系统版本，第一版不提供配置界面。

**验收：** 高风险分母来自最新 run；失败、unknown、无证据、证据质量问题进入高风险；普通界面仅显示中文业务原因。

### 阶段 4：追加式人工复核与审计

实施 `review_snapshots`、字段变更事件、首次复核人、快速通过、修改、无效证据、批量操作和 reopen。旧 `review_decisions` 切为只读兼容。

**兼容起点：** 新写路径实际启用时开始旧表兼容计数。阶段 4 本身不计作后续兼容验收周期。

**验收：** 原 assessment 不变；操作原因约束正确；sequence 连续；并发冲突返回 409；审计可还原原值、新值、复核人、时间和原因。

### 阶段 5：仪表盘、三栏工作台与完整核查表

实施报告 dashboard、桌面三栏复核、窄屏视图切换、PDF 证据定位和 577 条完整核查表。

**旧表兼容周期 1：** 运行旧 `review_decisions` 只读查询、新快照 effective result 和历史映射回归。失败时兼容计数归零。

**验收：** 高风险工作流可完成；未保存修改有拦截；桌面和窄屏截图无重叠；内部实现字段不进入普通 UI。

### 阶段 6：整改任务

实施从 assessment 创建整改任务、状态流转、负责人、截止日期、备注和筛选。整改状态不得自动改变 requirement 结论。

**旧表兼容周期 2：** 重复执行阶段 5 的兼容测试并比较历史映射。连续两轮通过后，旧表达到可清理条件。

**验收：** 报告隔离正确；完成、取消、重开原因可审计；任务与 assessment 保持引用但不改变结论。

### 阶段 7：版本化输出与兼容清理

实施草稿和正式输出、版本递增、supersede、复核范围 manifest、Excel、PDF、打印网页和整改清单。

若旧 API 或旧表已经完成两个连续兼容周期，则在独立 migration/变更中清理。清理必须满足：

- 数据库已有备份或可复现导出；
- 旧记录和新快照行数、关联和关键字段映射一致；
- 前端、后端、测试、脚本和文档无旧入口调用；
- 清理 migration 的 upgrade/downgrade 行为已验证；
- 清理变更与输出功能实现分开记录。

**验收：** 草稿随时可生成且有草稿标识；高风险未完成时阻止正式输出；正式版本不可变；导出明确区分人工确认和系统待确认结果。

### 阶段 8：端到端验收与泛化验证

1. 覆盖上传、metadata 确认、577 分析、部分失败、风险队列、复核、整改、草稿和正式输出的 E2E。
2. 重跑 Envision 577 baseline，确认 verdict、evidence、page、quality flags 和 OCR/VLM 风险标记无非预期回退。
3. 使用 Goldwind 或另一企业报告做泛化验证，不新增报告专用 per-ID contract。
4. 运行完整后端测试、前端 typecheck、测试和 build。
5. 启动本地服务，用桌面和窄屏截图检查核心页面。
6. 更新 `README.md` 和 `docs/DEVELOPMENT.md` 的运行、迁移、测试和验收说明。
7. 生成最终人工验收包，明确已自动验证、待人工确认、遗留风险和兼容清理结果。

**正常停止点：** 阶段 8 全部自动门禁通过后停止，等待用户进行产品验收。此时不再自动扩展新阶段。

## 4. 每个阶段的统一自动门禁

每个阶段结束时必须依次执行：

1. 相关单元测试和 API 测试；
2. Alembic upgrade 验证及允许范围内的 downgrade 验证；
3. OpenAPI schema 生成和契约测试；
4. 前端 API 类型生成、typecheck、组件测试和 build；
5. Envision 577 定向回归；
6. `git diff --check` 和路径扫描；
7. 检查 disclosed/review status、omission note、页码和 quality flags 等既有全局约束；
8. 将结果写入 `tmp/product-closure-continuous/stage-N-gate.md`。

门禁通过后自动进入下一阶段，不等待逐阶段人工复核。

## 5. 提前停止条件

出现以下任一情况立即暂停，保留现场并向用户报告原因、影响和处理建议：

1. migration 可能丢失、覆盖或无法完整映射现有人工记录；
2. 旧 API 或旧表兼容测试失败，或两个周期的数据映射不一致；
3. 577 eligible requirement 数量改变，或 Envision baseline 出现非预期 verdict/source/page 回归；
4. 正式输出无法追溯到 run、snapshot、规则版本和复核范围；
5. 实现核心流程必须调用未经批准的外部模型、OCR 或 VLM；
6. 需要新增多租户、复杂权限、新技术栈或未批准的产品状态；
7. 泛化验证只能依赖新增报告专用 per-ID contract 才能通过；
8. 当前工作区中的用户改动与目标文件冲突，无法在不覆盖用户改动的情况下继续；
9. 同一阻塞经过三轮独立修复仍未通过相关测试；
10. 核心页面在桌面或窄屏出现无法通过局部调整消除的布局重叠；
11. 自动测试揭示需要用户决定业务口径，且现有设计文档没有答案。

普通测试失败、实现难度、阶段数量和脏工作区本身不构成停止条件；应先定位并修复。

## 6. 最终人工验收包

阶段 8 完成后提供：

- 8 个阶段的完成状态和测试证据；
- 数据库 revision、兼容周期和旧接口/旧表清理结果；
- 本地访问地址及桌面、窄屏关键页面截图；
- 上传至正式输出的端到端演示数据；
- Envision 577 回归结果和 Goldwind 泛化结果；
- 已知非阻塞风险；
- 建议提交拆分和提交信息，但不自动 commit/push。

人工验收重点：页面信息架构、风险队列工作效率、三栏复核体验、正式输出内容和“高风险复核已完成”表述是否符合产品目标。
